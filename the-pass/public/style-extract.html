一組圖，怎麼變成驗證過的風格

丟一組參考圖給 /style-extract，它怎麼把「風格」抽成可重用的 profile——而且證明它生得出來。這頁拆解它的 input、harness、output。
抽一個風格不難，難的是證明真的抽到了。/style-extract 的特別之處在這：一個總指揮（orchestrator）親眼把風格讀成規格，接著生圖測試，再交給獨立的驗證 subagent 對著原圖逐面向打分——不是「看起來像」就算數，是分數證明能重現才算數。一句話：抽取靠整體的眼，驗證靠獨立的眼。
先解釋一個詞：harness（工法）。指這個 skill 不是「一個 AI 從頭做到尾」，而是一個總指揮（orchestrator）調度幾個各自獨立的分身（subagent）、中間設「停下來給人拍板」的關卡（gate）。像出菜口的廚房：主廚分派、各站位廚師各做各的（味道不互串）、出餐前主廚把關。下面整條流程就是這套工法。

整條流程

金色＝「停下來給人決定」的關卡；左邊那條迴路＝沒過就改了重來。

① 輸入：你要餵它的四樣

skill 開工前用 AskUserQuestion 一次問齊（這就是 GATE 0）。
要給的說明
一組參考圖5–15 張同調的圖最好。插畫、照片、社群圖、截圖、mood board 都能抽。
給什麼用插畫／照片／社群視覺——影響怎麼下提示詞、怎麼驗。
profile 名字之後別的專案就用這名字呼叫它（例：bold-pop、mono-ink）。
硬限制有沒有「一定要保留」的元素，或絕不要出現的東西。

② 為什麼特別：抽取靠整體的眼，驗證靠獨立的眼

這是整個 harness 的核心判斷——subagent 只用在「需要獨立」的地方，不是每件事。

很多人會直覺把「每個面向派一個 subagent 去分析」。我們刻意不這樣。風格是整體的：配色、線條、質感是一起作用的，拆成六個分身各看一塊，反而把它看碎、又慢又貴。所以抽取由 orchestrator 親眼一次讀完。

但驗證不一樣——它需要一雙不偏私的眼睛。寫 profile 的人（orchestrator）會對自己的作品「看起來覺得對」，容易給人情分。換一個沒參與寫 profile 的 subagent，純粹對著參考圖打分，才抓得到作者眼睛漏掉的飄移。所以驗證才拆 subagent，甚至可以多個獨立投票。

抽取（分析）驗證（打分）
誰做orchestrator 一次做每張測試圖一個獨立 subagent
為什麼這樣切風格是整體的，拆開反而看碎、又慢又貴需要不偏私的眼——作者會給自己人情分
動作六面向讀成規格對著參考圖、逐面向打 0–5
設計原則：別為了 harness 而 harness。subagent 是用來「隔離」的工具——哪裡需要獨立、不互相污染的判斷，才用它。分析不需要隔離（同一雙眼看整體最準）；驗證需要隔離（要一個沒立場的裁判）。這跟 /write-issue 把四個編輯、總編各開獨立分身是同一套思路。

③ 抽取：從六個面向把風格讀成規格

orchestrator 親眼看圖，把「風格」這個模糊的東西，拆成六條看得見、寫得下的軸。
面向抽什麼
palette 配色主色＋近似 hex、平塗還是漸層、背景怎麼處理
line 線條粗細、手感、向量／手繪／無線
texture 質感印刷感、顆粒、網點、筆觸
composition 構圖主體／留白／比例／視角／構圖語言
figuration 造形怎麼畫人與物、臉、比例、抽象程度
mood 情緒調性、氣場、聯想

這六條被寫進 style.md，再加上一段「每次生圖貼的提示詞片段」和一份 negative——這就是規格部分（profile 的 tell）。

④ 驗證：同樣的面向，交給獨立 subagent 打分

這是這個 skill 真正的價值。抽完不算數，要能生得出來、而且分數證明像，才算數。

用 profile 生 3 張不同主題的測試圖後，每張圖各派一個獨立的驗證 subagent，給它「該測試圖 ＋ 參考圖 ＋ 規格」，請它對著原圖逐面向打 0–5 分、標出哪裡飄掉、給一句總評。三份合起來＝重現性報告。

注意這裡的對稱：抽取用哪六個面向，驗證就用同一套面向打分。你沿著這些軸把風格讀出來，就沿著同樣的軸檢查它有沒有被重現。

真實案例：眼睛會被騙，分數不會。抽 mono-ink（毛筆墨線風）那次，肉眼看第一版「一次就很像」——但驗證 subagent 一打分，line 面向三張全是 2／5：漏了原作最關鍵的「毛筆粗細變化＋大實心黑塊」，畫成了均勻細線。把這點寫死進提示詞、重生重驗後，line 跳到 4–5，兩張升上 reproduced。沒有驗證，這個漏洞就直接變成 profile 的內傷了。

⑤ 三道 gate：人在哪裡拍板

全程互動，每道 gate 都用 AskUserQuestion——AI 跑、人決定，跟選題、寫稿同一個哲學。
GATE 0開工前 → 問齊四樣輸入（參考圖／用途／名字／硬限制）。
GATE 1草稿規格寫好 → 停，給你看：抓到風格了嗎？哪個面向要補？錨點選這幾張 OK 嗎？
GATE 2測試圖＋重現性報告出來 → 停：都 OK／某張飄了／整體不夠像。不夠像就回頭改提示詞、換錨點，重生重驗，直到穩定或你喊停。

⑥ 輸出：一個驗證過的 profile

三部分落在磁碟上 = 一個 style.md ＋ 兩個資料夾。完整拆解見 換插圖風格 SOP。
~/style-lab/profiles/<名字>/ ← 跨專案共用，任何專案來消費 ├── style.md 規格（六面向）＋ 提示詞片段 ＋ negative ＋ 驗證摘要　the tell ├── anchors/ 你給的原圖（複製進來，2–4 張）　　　　　　　　the show └── validation/ 生的測試圖 ＋ report.md（逐面向打分）　　　　the prove

各專案的出圖 skill（如 The Pass /publish-issue）去讀選定的 profile 來生圖。換風格＝換專案指向哪個 profile，生圖流程本身永遠不動。

⑦ 這個 skill 由什麼組成

skill 本體很薄（一份工作流），重量都在它產出的共用風格庫。
~/.claude/skills/style-extract/ ← skill 本體（跨專案、使用者層級） └── SKILL.md 工作流：orchestrator 怎麼讀圖、3 道 gate、驗證 subagent 怎麼打分、迭代迴圈 ~/style-lab/ ← 共用風格庫（skill 的產出歸宿，不碰任何專案目錄） ├── README.md └── profiles/ ├── bold-pop/ style.md ＋ anchors/ ＋ validation/ └── mono-ink/ 同上
為什麼放在專案外？抽出來的風格不只給電子報用——投影片、社群、客戶視覺都能套。所以 profile 存共用庫 ~/style-lab/，The Pass 只是它的消費者之一。測試圖也一律待在這裡，絕不寫進任何專案的 public/、絕不部署到網站（那是污染專案）。

守門原則

抽取靠整體的眼，驗證靠獨立的眼。分析由 orchestrator 一次讀完（風格是整體的）；驗證交給沒寫過 profile 的 subagent（要不偏私的裁判）。
看起來像 ≠ 真的重現。驗證要對著參考圖誠實打分、不給人情分；飄掉就說飄掉。交付「驗證過」的，比交付「看起來對」的可靠太多。
錨點圖 > 文字。純文字提示詞釘不住具體風格，錨點圖（餵進生圖模型當 input）才鎖得住——profile 一定要帶錨點。影片截圖／有字幕的，先裁乾淨再當錨點，否則模型會把字幕一起學進去。
抽風格、不抄人。抽的是配色／線條／質感／構圖語言；不照搬某位作者的角色或具體構圖。產出要是「自己的概念，那個風格」。
