01總覽

The Pass 的目標是建立一個半自動化的內容產出流程：每週二、五各產出一期，每期包含 2 篇長文、3 則快訊、1 個結尾提問、以及 2 張插圖。

整個系統要解決四件事：

  1. 從哪裡抓內容（信源監測）
  2. 怎麼選內容（篩選與選題）
  3. 怎麼寫內容（AI 編輯撰寫）
  4. 怎麼發內容（發佈與分發）

你每天實際操作時間的目標：15-20 分鐘。

剩下的全部交給自動化。你的時間應該花在品味校準和最終確認上，不是重複性操作。

02系統架構

以下是推薦的技術 stack。每一層工具的選擇都基於同一個原則：side project 等級的成本、production 等級的能力。

推薦架構

功能工具用途
信源抓取 + 排程Vercel Cron Jobs每天自動跑 RSS 抓取
資料儲存Supabase存信源、原始文章、已發佈期刊、訂閱者
AI 內容產製Claude API翻譯、評分、撰寫、總編輯審核
發佈平台Ghost（建議 Ghost Pro）網站 + 電子報一體化
插圖生成AI 圖片生成 API每期 2 張插圖
日常操作待定（見 Section 06）你每天操作的介面

為什麼選 Ghost

  • 網站 + 電子報 + 會員系統一體化 -- 不需要用三個工具拼湊
  • 完整的 Admin API，可接自動化
  • 品牌完全自訂，讀者看不到第三方標誌
  • 原生支持 Stripe 付費訂閱（未來變現）
  • 開源、社群強、生態成熟

Ghost Pro vs 自架

Ghost Pro（託管）自架（Fly.io / Railway）
月費$9（500 會員）~$3-5
維護零，官方管理自己管更新、備份、SSL
更新自動手動
支援官方客服社群
自訂Theme 層面完全掌控
建議Side project 首選適合有 DevOps 經驗且想省錢的人

結論：建議 Ghost Pro。$9/月買的是「不用管 server 的自由」。

03每期產出流程

從信源進來到讀者收到電子報，整個 pipeline 分五個階段。每個階段標明了「你需不需要參與」以及「花多少時間」。

全自動 Phase A -- 信源監測
時間：每天凌晨 6:00 自動執行 你的參與：無

Vercel Cron 觸發 API route，抓取所有信源的 RSS。AI 翻譯非英文摘要，並進行初步評分（相關性 x 新鮮度 x 趣味性），結果存入 Supabase。

產出：「今日原料清單」 -- 20-40 則候選內容，按分數排序。

半自動 Phase B -- 選題
時間：你早上打開操作介面的時候 你的參與：~10 分鐘

AI 已經推薦 Top 5-8 則候選。你瀏覽清單，勾選 2 則做長文、3 則做快訊（或調整 AI 推薦）。

產出：確認的選題清單。

備註：初期建議你多花一點時間自己看原料清單，校準 AI 的選題標準。等 AI 的推薦準確率夠高之後，你只需要 approve / reject。

全自動 Phase C -- AI 撰寫
時間：你確認選題後立即執行 你的參與：無（等待 2-3 分鐘）
  • Mise（主筆）根據選中的素材寫 2 篇長文（~500 字/篇）
  • Passe（快訊編輯）寫 3 則快訊（~150 字/則）
  • Fumet（提問者）寫結尾問題（如果這期需要）
  • 每位編輯使用各自的 System Prompt
  • 同時觸發插圖生成（2 張，使用風格指南的 prompt 模板）

產出：完整的一期草稿。

半自動 Phase D -- 審核
時間：Phase C 完成後 你的參與：~5 分鐘

AI 總編輯自動 review 草稿，檢查事實核查、AI 味檢測、標題品味、整期節奏、品牌一致性。

  • 如果有問題：標記具體位置 + 建議修改，AI 編輯自動修正，再過一次總編輯
  • 如果通過：顯示「可發佈」
  • 你做最終確認

產出：通過審核的完稿。

全自動 Phase E -- 發佈
時間：你按下「發佈」按鈕後 你的參與：按一個按鈕
  • 透過 Ghost Admin API 建立文章 + 設定排程
  • Ghost 自動發 email 給所有訂閱者
  • （可選）同時輸出 IG 適用的版本

產出：網站更新 + 電子報發出。

04AI 角色定義

整個 pipeline 中涉及七個 AI 角色，各自在不同階段介入。

角色功能何時介入
翻譯引擎將韓文/日文/泰文信源翻譯成中文摘要Phase A
評分引擎對候選內容進行相關性/新鮮度/趣味性評分Phase A
Mise（主筆）撰寫今日觀察長文Phase C
Passe（快訊編輯）撰寫快訊Phase C
Fumet（提問者）撰寫結尾問題Phase C
插圖生成根據文章內容 + 風格指南生成插圖Phase C
AI 總編輯最終品質審核Phase D

AI 總編輯的審核清單

  • 事實核查 -- 數字、公司名、人名是否正確
  • 標題品味 -- 是否符合「先找到人，再提到技術」
  • AI 味檢測 -- 是否有條列式、禁止詞彙、機器腔
  • 重複檢查 -- 5 則內容主題是否太相似
  • 整期節奏 -- 長短交錯、國家多元
  • 品牌一致性 -- 語氣是否符合 The Pass 調性

05自動化分級

不是所有事情都應該自動化。以下是清楚的三層分級，讓你知道哪些環節需要你介入、哪些完全不用管。

全自動（不需要人）

  • RSS 信源抓取 + 翻譯 + 評分
  • AI 撰寫文章
  • AI 生成插圖
  • AI 總編輯 review
  • 排版組裝成一期
  • 電子報排程發送

半自動（AI 做，人確認）

  • 選題（AI 推薦，你 approve / reject）
  • 最終發佈（你按一個按鈕）

手動（只有人能做）

  • 品味校準（定期調整選題標準和 prompt）
  • 信源 review（每月/每季評估信源品質）
  • 風格微調（覺得語氣或插畫不對時）
  • 特殊企劃（年度回顧、專題策劃）
  • 新信源發掘

06操作介面選項

目前有兩種方案，各有優劣，尚未決定。第三種是我們推薦的漸進路線。不論選擇哪一種，底層的程式邏輯是相同的——差別只在「你用打字操作，還是用點按鈕操作」。

方案 A

Claude Code Skills

將每個工作步驟包裝成 Skill（slash command），在 Claude Code 裡操作。5 個 Skill 涵蓋完整流程。

Skill背後做的事你看到的
/fetch-sources 跑 RSS 抓取器 → AI 翻譯非英文摘要 → AI 評分（相關性、新鮮度、趣味性）→ 存入 Supabase 「今天抓到 34 則候選，AI 推薦 Top 6」+ 清單
/select-stories 讀取 Supabase 的候選清單 → 顯示標題、摘要、分數 → 等你選擇 → 將選中的標記為「本期素材」 候選清單，你說「選 1、3、5 做長文，2、4 做快訊」
/generate-issue 依序呼叫 4 個角色：
1. Mise 用他的 system prompt 寫 2 篇長文
2. Passe 用他的 system prompt 寫 3 則快訊
3. Fumet 用他的 system prompt 寫結尾提問
4. 圖片 API 用風格指南 prompt 生成 2 張插圖
全部完成後組裝成一期完整草稿		 「生成中... Mise 寫完 → Passe 寫完 → Fumet 寫完 → 插圖完成。草稿已就緒。」
/review-issue AI 總編輯讀取草稿 → 逐項檢查（事實核查、標題品味、AI 味檢測、重複、節奏、品牌一致性）→ 通過或標記問題 「總編輯審核：通過 5/6 項。標題第 3 則建議修改，原因：太像效率切入。」→ 自動修正 → 再審一次 →「全部通過，可發佈。」
/publish 呼叫 Ghost Admin API 建立文章 → 設定排程 → Ghost 自動寄 email → 更新網站 「Issue #015 已發佈。網站已更新，電子報將於 09:00 發送給 342 位訂閱者。」

注意：/generate-issue 不是四個人四個 Skill，而是一個 Skill 裡面依序叫四個角色。就像餐廳裡你喊一聲「出菜」，前菜師傅、主菜師傅、甜點師傅各自動手，最後端出一整桌菜。

優點

  • 每個 Skill 半小時就能建好，整套流程一天內可以跑起來
  • 極度靈活——覺得哪裡不對，改一個 markdown 檔就好
  • 不需要寫前端，Claude Code 本身就是操作介面
  • 可以隨時插入人工判斷（例如「這篇幫我改一下標題」）

缺點

  • 只有你（或有 Claude Code 的人）能操作
  • 沒有視覺化介面，全靠文字互動
  • 看候選清單時不如有 UI 的版本直覺
  • 團隊協作困難

適合：Phase 0-1，一個人跑流程，快速驗證。

方案 B

Web 後台（Admin Panel）

在 Vercel 專案裡建一個 /admin 頁面，用瀏覽器操作。骨子裡跑的程式碼和 Skill 完全一樣，只是前面多了一層視覺介面。

Skill 版（打字）Web 後台版（點按鈕）骨子裡跑的邏輯
打 /fetch-sources 頁面自動顯示「今日原料清單」 同一個 RSS 抓取 + 評分函式
打 /select-stories
說「選 1, 3, 5」		 在清單上勾選 → 按「確認選題」 同一個 Supabase 更新函式
打 /generate-issue 按「生成本期內容」
看到進度條		 同一個 Claude API 呼叫鏈
打 /review-issue 審核結果自動顯示在預覽頁 同一個總編輯 prompt
打 /publish 按「發佈」按鈕 同一個 Ghost API 呼叫

優點

  • 視覺化介面——候選清單一目了然，勾選比打字快
  • 團隊成員都能用，不需要會 Claude Code
  • 可以做更精細的 UI（即時預覽、歷史紀錄、數據儀表板）
  • 排程功能——設定好之後自動跑，不用每天手動觸發
  • 更像一個「產品」

缺點

  • 需要 1-2 週開發
  • 改流程要改程式碼，不像 Skill 改 markdown 就好
  • 在流程還沒穩定之前就建 UI，很容易做白工

適合：Phase 2 以後，流程穩定，需要團隊協作。

方案 C — 推薦

混合（漸進路線）

Phase 0-1 先用 Skill 跑起來，驗證整個流程。等到知道「每天需要什麼操作」之後，再把 Skill 的邏輯搬到 Web 後台。

階段操作方式目的
Phase 0-1
（Week 1-4）		 Claude Code Skills 快速驗證流程、校準選題品味、調整 prompt。每天實際操作，發現哪些步驟需要人介入、哪些可以全自動。
Phase 2
（Week 5-8）		 Skill → API routes 把 Skill 裡的核心邏輯抽出來，變成 Vercel 的 API routes。Skill 改成呼叫這些 API，而不是自己跑邏輯。這一步不影響你的操作方式。
Phase 3
（需要時）		 加上 Web UI 在 API routes 前面套一個 /admin 介面。因為你已經用 Skill 跑了幾週，非常清楚每個頁面、每個按鈕該做什麼。不會做白工。

Skill 不是臨時方案——它是建後台最好的 prototype。跑順了之後，你已經知道後台該長什麼樣了。

08實作時程

WEEK 1基礎建設
  • 註冊 Ghost Pro，設定 The Pass 品牌頁面
  • 建立 Supabase 資料庫 schema
  • 寫第一版 RSS 抓取腳本（先接 5 個核心信源）
  • 設定 Vercel Cron Job
WEEK 2手動驗證
  • 每天看「原料清單」，手動選題
  • 手動寫 2-3 期 The Pass（用 Ghost 後台直接寫）
  • 驗證：每天有沒有足夠的素材？選題標準對不對？
  • 校準 AI 評分的權重
WEEK 3接入 AI 編輯
  • 將 Mise / Passe / Fumet 的 System Prompt 接入 Claude API
  • 讓 AI 開始撰寫，你做最終修改
  • 比較 AI 寫的和你手動寫的差異，調整 prompt
WEEK 4加入自動化
  • 建立 AI 總編輯 review 機制
  • 串接 Ghost Admin API，實現一鍵發佈
  • 加入插圖自動生成
  • 建立操作 Skill（或開始規劃 Web 後台）
WEEK 5-6優化
  • 根據前 4 週的經驗調整所有 prompt
  • 優化選題評分算法
  • 建立信源品質監測
  • 開始正式對外發佈

09內容再利用（Re-purpose）

每一期 The Pass 產出的內容是「原料」，可以被轉換成不同平台的格式。每個輸出管道就是一個獨立的 Skill（或 Web 後台上的一個按鈕）。

架構邏輯

/generate-issue 產出的一期完整內容，是所有管道的共同原料。每個 re-purpose Skill 讀取同一份完成稿，根據該平台的格式和限制重新編排。內容不重寫，只換呈現方式。

Skill輸出平台做的事優先級
/publish Ghost 網站 + 電子報 主要發佈管道。透過 Ghost Admin API 建立文章，Ghost 自動寄 email。 核心（Week 1）
/to-instagram Instagram 從長文中挑 1 篇最適合 IG 的 → 500 字壓縮成 150 字 caption → 插圖裁切成 1:1 圖卡 → 自動加 hashtags → 產出圖 + caption 待你確認 Phase 2
/to-threads Threads 將一期內容拆成 3-5 則串文 → 每則 100 字以內 → 第一則是 hook → 最後一則導流到完整版 Phase 2
/to-line LINE 推播 壓縮成 LINE 適合的短訊格式 → 一則主標題 + 一句摘要 + 連結 → 配合 LINE OA 發送 Phase 3
/to-x X（Twitter） 每則快訊變成一則推文 → 280 字以內 → 長文變成一個 thread → 附原文連結 Phase 3

執行原則

先把 Ghost 這條主線做穩。/publish 是唯一的核心 Skill，其他 re-purpose Skill 都是「加分項目」，等主流程跑順之後再逐一加入。每多一個管道就多一個觸及點，但每個管道的邊際成本幾乎為零——因為內容已經在那裡了，只是換一種方式送出去。

一份內容，多次觸及。這是 The Pass 作為 AI 驅動內容產品的結構性優勢——人力成本不會因為管道數量增加而等比成長。
