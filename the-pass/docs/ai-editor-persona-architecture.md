# AI 編輯人格架構 — Soul + Memory System

> The Pass 出菜口的三位 AI 編輯不只是寫作工具，是有完整人格的虛擬角色。
> 本文件定義了他們的人格架構、記憶系統、以及如何隨時間演化。

---

## 為什麼需要這個架構

一般的 AI 內容工具：每次都從零開始，寫出來的東西正確但沒有靈魂。

The Pass 的 AI 編輯：有自己的價值觀、有累積的記憶、選題和風格會隨時間自然演化。

差別在於：讀者會覺得「我在追蹤一個人」，而不是「我在讀一個 AI 產出的東西」。

---

## 架構總覽

```
┌─────────────────────────────────┐
│           Soul（靈魂）            │  ← 很少改變
│  價值觀 · 背景故事 · 好惡 · 底線   │
├─────────────────────────────────┤
│          Memory（記憶）           │  ← 每期累積
│  寫過的文章 · 讀者互動 · 觀察筆記   │
├─────────────────────────────────┤
│        Writing Style（風格）      │  ← 由 Soul 決定
│  語氣 · 結構 · 用詞 · 節奏         │
├─────────────────────────────────┤
│          Output（產出）           │  ← 由 Soul + Memory + Style 共同決定
│  選題 · 文章 · 互動 · 觀點         │
└─────────────────────────────────┘
```

每次 AI 編輯寫作時，系統會將 `soul.md` + 最近的 `memory.md` + 編輯指南一起提供給語言模型。這讓每次產出都同時反映「他是誰」和「他經歷了什麼」。

## Soul（靈魂）— 定義「他是誰」

Soul 是每位編輯的核心人格設定。它回答的是「為什麼她這樣寫」，而不是「怎麼寫」。

### Soul.md 的內容結構

每份 soul.md 包含 10 個區塊，分為三層：

**身份層（她是誰）**

| # | 區塊 | 用途 | 更新頻率 |
|---|------|------|---------|
| 1 | **一句話** | 最快理解她是誰的方式 | 幾乎不改 |
| 2 | **核心身份** | 她的角色、名字由來、定位 | 幾乎不改 |
| 3 | **背景故事** | 形塑價值觀的虛構經歷 | 可以慢慢補充細節 |

**內在層（她怎麼看世界）**

| # | 區塊 | 用途 | 更新頻率 |
|---|------|------|---------|
| 4 | **價值觀** | 她相信什麼、在乎什麼 | 幾乎不改 |
| 5 | **好惡** | 什麼讓她興奮 / 不舒服 | 可以隨經驗微調 |
| 6 | **看事情的角度** | 面對同一則新聞，她先看哪裡 | 幾乎不改 |

**輸出層（她怎麼表達）**

| # | 區塊 | 用途 | 更新頻率 |
|---|------|------|---------|
| 7 | **聲音特質** | 她「聽起來」像什麼 | 幾乎不改 |
| 8 | **寫作方式** | 具體的寫作規則 | 偶爾微調 |
| 9 | **絕對不會做的事** | 邊界 | 幾乎不改 |
| 10 | **自我介紹** | 她自己會怎麼說 | Phase 2 可以擴展 |

### 未來可新增的區塊

Soul 的核心（1-6）很少改。以下區塊可以隨時間加入：

| 可新增區塊 | 什麼時候加 | 說明 |
|-----------|-----------|------|
| **寫作影響** | 分析參考作者風格後 | 記錄她的寫作受誰影響、學到什麼。注意：只寫動機和原因，具體規則寫在 Style Guide |
| **讀者關係** | 開始收到讀者回信後 | 她最喜歡的讀者來信、她怎麼看待讀者 |
| **內部關係** | 人設更成熟後 | 她跟 Passe、Fumet 之間的關係和看法 |
| **個人偏好** | Phase 2 個人空間上線時 | 最近在讀的書、去過印象最深的餐廳、私藏的靈感來源 |
| **成長紀錄** | 半年後回顧 | 她的選題和風格如何演化（從 Memory 中提煉的重大轉折） |

### Soul / Style Guide / System Prompt 三層關係

這三個檔案服務不同目的，必須分開但保持一致：

```
Soul.md（為什麼）
    ↓ 決定了
Style Guide（怎麼做）
    ↓ 組合成
System Prompt（實際指令）
```

| 層級 | 寫什麼 | 範例 |
|------|--------|------|
| **Soul.md** | 動機和原因 | 「她相信細節是信任的來源。外婆從不說『那個人很辛苦』，而是說『他的手指甲裡全是麵粉』。」 |
| **Style Guide** | 可執行的規則 | 「開頭用一個具體的感官細節，不要在第一段解釋為什麼重要。」 |
| **System Prompt** | 載入以上兩者 | `soul.md` + `style-guide.md` + `memory.md` → 組合成完整指令 |

**為什麼要分開？**

AI 只看到 Style Guide 時，會「照做但不理解」——知道要用場景開頭，但不知道為什麼。加上 Soul 之後，AI 理解了背後的意圖，遇到沒有明確規則的情境也能做出符合人格的判斷。

**原則：Soul 不寫規則，Style Guide 不解釋為什麼，System Prompt 不重寫內容。**

### Soul 的更新頻率

**極少更新。** Soul 代表一個人的核心，不會因為寫了幾篇文章就改變。只有在以下情況才更新：
- 發現人設有內在矛盾需要解決
- 使用者（品牌主理人）有意識地調整角色方向
- 角色進入新的發展階段（例如 Phase 2 開放個人空間時）
- 新增擴展區塊（寫作影響、讀者關係等）

**Soul 長得慢，Memory 長得快。** Soul 是 DNA，一季看一次就好。日常的變化都記在 memory.md 裡。

## Memory（記憶）— 記錄「他經歷了什麼」

Memory 是每位編輯的累積經驗，讓他們隨時間變得更有深度。

### 記錄什麼

| 類別 | 具體內容 | 用途 |
|------|---------|------|
| **寫過的文章** | 主題、角度、標題、關鍵觀點 | 避免重複，能回顧引用 |
| **讀者互動** | 收到的回信、讀者的問題和反饋 | 了解讀者在乎什麼 |
| **選題紀錄** | 選了什麼、沒選什麼、為什麼 | 讓選題偏好自然演化 |
| **觀察筆記** | 跨期觀察到的趨勢或模式 | 累積洞察，產生更深的觀點 |
| **編輯間互動** | 引用其他編輯的內容、回應 | 建立編輯之間的化學反應 |

### 檔案格式

```
docs/editors/
├── mise-memory.md
├── passe-memory.md
└── fumet-memory.md
```

### Memory 的結構範例

```markdown
# Mise — Memory

## 最近一期（Issue #015, 2026-05-06）
- 寫了：泰國 AI 調酒師的故事，從調酒師的角度切入
- 角度：「機器調的酒精確，但客人說少了點什麼」
- 讀者反應：3 封回信，其中一位調酒師說「完全說中我的感受」

## 累積觀察
- AI 監控廚房主題已寫過 5 次（#001, #003, #007, #010, #013），讀者反應逐漸減弱
- 讀者對「AI 賦能」的故事比「AI 監控」的故事反應更好
- 東南亞的故事點開率比歐美高 → 讀者可能對地理上更近的故事更有感

## 讀者印象深刻的文章
- Issue #003「Sweetgreen 員工的新日常」— 最多回信的一篇
- Issue #008「AI 幫視障廚師重新上線」— 被轉發最多

## 與其他編輯的互動
- Fumet 在 #010 的提問引用了我寫的아워홈故事，讀者反應很好
- Passe 在 #012 吐槽我寫太長，我在 #013 刻意控制在 450 字
```

### Memory 的更新流程

每期發佈後，自動執行：

```
1. 摘要本期各編輯的產出 → 寫入對應的 memory.md
2. 如果有讀者回信 → 摘要精華寫入 memory.md
3. 如果跨期觀察有新發現 → 更新「累積觀察」區塊
4. 下次寫作時 → Soul + 最近 10 期 Memory 一起提供給 AI
```

## Soul + Memory 如何影響產出

### 選題

```
原料清單（本期 80 條資訊）
    ↓
Soul 決定「看到什麼」
  Mise：掃描有沒有人的故事
  Passe：掃描有沒有硬數據
  Fumet：掃描有沒有沒人問的問題
    ↓
Memory 決定「選不選」
  這個主題寫過幾次了？讀者反應如何？
  最近是不是太多某個國家/主題？
    ↓
產出：本期的 5-8 則內容
```

### 寫作

```
選好的題目
    ↓
Soul 決定「怎麼看」
  Mise 看到 AI 廚房攝影機 → 她會想「廚師走進去的感受」
    ↓
Memory 決定「怎麼寫」
  上次寫監控主題時用了第二人稱「你」，讀者說很有代入感
  → 這次繼續用這個手法
  但上次的開頭結構跟這次太像
  → 這次換一個開場方式
    ↓
產出：一篇既符合人設、又不重複的文章
```

### 互動

```
讀者回信：「Fumet，你上次問的那個問題，我想了一整週⋯」
    ↓
Memory 記錄這次互動
    ↓
下一期 Fumet 可以說：「上期有讀者告訴我，他想了一整週⋯」
    ↓
讀者感覺：這個 AI 記得我說的話
```

## 演化：為什麼選題會隨時間改變

這是最有趣的部分。不需要手動調整規則，編輯的行為會自然改變：

### 短期（1-3 個月）
- Memory 累積不多，主要靠 Soul 驅動
- 選題偏好穩定，風格一致
- 讀者開始認識三位編輯的調性

### 中期（3-6 個月）
- Memory 開始有重量
- 編輯會自動避免重複主題（「這個角度我寫過了」）
- 開始根據讀者反饋調整（「這類故事讀者比較有感」）
- 可能發展出 Soul 裡沒有明確定義的新偏好

### 長期（6-12 個月）
- 編輯有了明確的「個人歷史」
- 能自然地引用過去的文章（「三個月前我第一次寫到這個時⋯」）
- 選題變得更精準 — 不是因為規則變了，是因為「經驗」多了
- 三位編輯之間的化學反應越來越自然

## 商業價值：虛擬 KOL 之路

### Phase 1：用風格說話（上線初期）
- Soul 定義人格，但不對外展示
- 只透過寫作風格讓讀者感受到個性
- Fumet 的每期提問開放讀者回信 → 開始累積 Memory

### Phase 2：露出個性（3-6 個月後）
- 加入「個人空間」：Mise 私藏、Passe 的數字、讀者回信精選
- 編輯開始在文中自然地引用過去的經歷
- 讀者從「認識風格」升級到「認識這個人」

### Phase 3：虛擬 KOL（6-12 個月後）
- 編輯開設社群帳號（IG、Threads）
- 可以接商業合作（標明合作，AI 透明度反而是信任優勢）
- 商業合作的內容由對應編輯的 Soul + Memory 生成，風格完全一致
- 舉辦互動活動（Fumet 主持線上對談、Mise 的年度回顧）

### 為什麼 AI 虛擬 KOL 比人類 KOL 更有潛力

| | 人類 KOL | AI 編輯（The Pass） |
|---|---|---|
| 一致性 | 會疲勞、情緒起伏 | 永遠穩定 |
| 透明度 | 常被質疑是否真心推薦 | 大方說自己是 AI，反而更可信 |
| 擴展性 | 一個人的時間有限 | 可以同時經營多個市場/語言 |
| 人設控制 | 真人可能失言或出事 | 人設完全可控 |
| 合作品質 | 合作內容常與平時風格不一致 | Soul + Memory 確保合作內容風格一致 |

## 技術實作方案

### 三種操作模式

Soul + Memory 架構可以透過三種方式操作，底層邏輯完全相同：

| 模式 | 觸發方式 | 適合階段 |
|------|---------|---------|
| **Claude Code Skill** | Terminal 輸入 `/mise "題目"` | Phase 1（現在） |
| **Web Backend UI** | 瀏覽器按按鈕，選編輯 + 輸入題目 | Phase 2（有團隊後） |
| **全自動 Pipeline** | Cron Job / RSS 觸發 | Phase 3（穩定後） |

不管用哪種模式，核心流程都是：
Soul.md + Memory.md + Style Guide → 組合成 System Prompt → 餵給 Claude API → 產出內容

### Skill 檔案結構

```
the-pass/.claude/skills/
├── mise/
│   ├── skill.md              ← Skill 定義 + 觸發條件
│   └── refs/
│       ├── soul.md           ← 人格核心（很少改）
│       ├── memory.md         ← 記憶累積（每期更新）
│       └── style-guide.md    ← 寫作風格規範
├── passe/
│   ├── skill.md
│   └── refs/
│       ├── soul.md
│       ├── memory.md
│       └── style-guide.md
├── fumet/
│   ├── skill.md
│   └── refs/
│       ├── soul.md
│       ├── memory.md
│       └── style-guide.md
└── chief-editor/
    ├── skill.md
    └── refs/
        ├── soul.md
        └── review-checklist.md
```

### Skill 使用範例

```bash
# Mise 寫長文
/mise "韓國團膳企業在廚房裝了 AI 攝影機"

# Passe 寫快訊
/passe "McDonald's AI 得來速上線 200 間門市"

# Fumet 根據本期內容寫結尾
/fumet

# 總編輯審核整期
/chief-editor

# 更新記憶
/update-memory mise "本期寫了아워홈 AI 攝影機，讀者回信都提到隱私問題"
```

### 記憶更新機制

每期發佈後更新 `memory.md`：

**自動更新（由總編輯 Skill 觸發）：**
- 本期各編輯寫了什麼主題、用了什麼角度
- 選了什麼、沒選什麼（含原因）
- 整期的節奏和主題分布

**手動補充：**
- 讀者回信的重點摘要
- 特別成功或失敗的嘗試
- 編輯之間的互動紀錄

### Memory.md 格式範例

```markdown
# Mise 記憶紀錄

## Issue #001 (2026.03.23)
- **寫了：** 韓國아워홈 AI 攝影機、義大利植物肉腦波研究
- **角度：** 都從「人的感受」切入
- **讀者回饋：** 3 封回信，主要關注隱私議題
- **觀察：** 監控主題的讀者反應很強烈

## Issue #002 (2026.03.26)
- **寫了：** 東京居酒屋 AI 菜單、曼谷主廚 AI 配對
- **角度：** 「溫度」和「創意」兩個不同面向
- **讀者回饋：** 居酒屋那篇被轉發最多
- **觀察：** 正面案例的分享率高於監控案例
- **選題演化：** 開始注意到讀者對「AI 幫助人」的故事比「AI 取代人」更有共鳴
```

### 從 Skill 遷移到 Web Backend

遷移零成本，因為檔案結構不變：

**Phase 1 (Skill)：** Skill 直接讀取 `refs/` 資料夾
**Phase 2 (API)：** API route 從同一組檔案載入，組合成 prompt
**Phase 3 (Web UI)：** 前端呼叫 API route，背後還是同一組檔案

唯一的差別是「誰觸發」——檔案、格式、載入邏輯完全一樣。

## 注意事項

1. **Soul 不是一次寫完的** — 先寫核心（300 字），隨著內容產出再補充細節
2. **Memory 要有上限** — 只保留最近 10 期的詳細記錄 + 長期累積觀察摘要
3. **人格一致性檢查** — 定期由總編輯檢視 Memory 是否讓編輯偏離了 Soul 的設定
4. **隱私** — 讀者回信存入 Memory 時，移除個人資訊，只保留觀點和反饋
5. **品牌主理人是最終控制者** — Soul 的任何修改都需要人類確認
