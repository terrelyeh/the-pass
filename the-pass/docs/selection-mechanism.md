# 篩選機制設計 — The Pass 選題流程

> 從「原料進來」到「選出一期」的完整 pipeline。
> 對應程式：[`src/lib/sources.ts`](../src/lib/sources.ts)、[`src/lib/fetcher.ts`](../src/lib/fetcher.ts)
> 相關：[source-verification-checklist.md](source-verification-checklist.md)、memory: project_sourcing_angles

---

## 0. 一句話心法

> **「抓取」是每天累積的，「選題」是出刊前才做的，中間靠一個有記憶的池子隔開。**

不是「抓→評→出」一次跑完，而是一條有狀態、會累積、會回饋的流水線。

---

## 1. 全流程總覽

```
每日抓取（全部 active 來源，統一頻率）
      │
      ▼  去重 ① URL 完全相同   ② 語意 同則新聞不同來源
┌─────────────────────────────────┐
│  標題池 Pool（累積 + 有狀態）        │  狀態：新 / 已評分 / 已出刊 / 已過期
└─────────────────────────────────┘
      │  ← 出刊前才跑（一週 2 次：週二、週五）
      ▼  ① Opus 評估（硬閘門 + 五面向，整池每篇都看）→ ② 排序 + 組成一期
   AI 初選：組好整期推薦 + 每篇主理由
      │
      ▼  你拍板（approve / 換 / 否決 / 撈庫存）
   本期選用（~2 長文 + ~3-6 快訊 + Fumet 提問）
      │
      ├─▶ 終選回寫 → 三位編輯 memory（越用越貼品味）
      └─▶ 其餘合格 → 庫存 backlog（帶保鮮期）→ 下期重新競爭 / 餵 IG / 月報
```

## 2. 取材：兩條進料線（兩種角度）

| | Stream A | Stream B |
|--|----------|----------|
| 路徑 | **飲食媒體 → 找 AI** | **AI 前沿 → 找飲食** |
| 抓到的故事 | 「某餐廳導入了 AI」 | 「某新 AI 能力將改變餐飲」 |
| 特性 | 已落地、有人味、在地 | 早、技術、全球 |
| 來源例 | The Spoon、Eater、식품외식경제 | Ben's Bites、Synced、HF Papers |
| 額外要求 | — | 食物訊號密度低（淘金）+ 需重度編輯翻譯 |

> Stream B 是「低產量、高價值的礦」，不是「穩定溪流」。撈到的前沿研究（如 Inverse Cooking）必須先翻成人話（「拍一張阿嬤的菜，AI 還原失傳食譜」），翻不出人味就砍。

---

## 3. 抓取頻率

- **統一每日抓取一次全部 active 來源。** RSS 成本極低，一週出 2 期，每日抓足夠有餘。
- 不為不同來源排不同 cron——那是還沒需要就先增加的複雜度。
- *何時才差異化？* 等接了有 rate limit 的 API、或有即時性極強的來源時再說。
- HF Papers（`feedType: json`）天然每日一次，與此節奏吻合。

---

## 4. 去重（pipeline 的「記憶」）

RSS 每次抓都回傳同一批最近文章，沒有去重就會重複選、重複出。需要一個 **seen store**，分兩層：

1. **完全相同（同一篇）**：用文章 **URL 當唯一鍵**。抓到已看過的直接丟，只有新文章進池子。
2. **同則新聞、不同來源**：例如 Chef Robotics 融資被 The Spoon 和 TechCrunch 同時報——URL 不同但同一件事。靠**標題語意相似度**去重，避免選到兩篇幾乎一樣的。

> 初期可用一個 JSON 檔當 seen store，正式版接 Supabase。

### ⚠️ 待修 Bug

| Bug | 位置 | 問題 | 修法 |
|-----|------|------|------|
| 去重失效 | `fetcher.ts` 文章 id | id 用 `${source.id}-${isoDate}-${Date.now()}`，`Date.now()` 讓同一篇每次抓都產生不同 id，去重根本無效 | 改用 **URL** 當穩定唯一鍵 |
| AI 假陽性 | `fetcher.ts` `scoreRelevance` | 用 `text.includes("ai")` 判斷，會誤中 `t**ai**l`、`av**ai**lable`、`ag**ai**n`，把純餐廳清單判成「AI×食物」 | 改詞邊界比對 `\bai\b`，或移除裸 `"ai"` 只留 `"a.i."` / `"artificial intelligence"` |

## 5. 篩選：Opus 全程評估

去重後的**每一篇**都直接交給 Opus 做一次完整判斷——先過硬閘門、再打五面向，**同一個模型、同一次呼叫**。不先用關鍵字粗篩、也不用便宜模型墊第一關。

> 為什麼不先用關鍵字省錢？關鍵字本質是個「會漏、會誤判的省錢代用品」（曾出現 `tail→ai` 假陽性、也會漏掉沒寫「AI」的食品科技，如培養肉）。真正決定成本的是**每日去重後的新篇數**（通常 ≤25），不是 feed 大小。池子既然夠小，就值得用最強模型整池全看——最一致、最不漏稿。成本上限靠去重壓小池子；若單次抓取異常爆量，再加「純數量」上限當斷路器即可，絕不退回關鍵字粗篩。

### 第一步 — 硬閘門（pass / fail，沒過直接砍）

每篇先過硬閘門再打「五面向」。閘門條件見 §6。

#### 硬閘門（pass / fail，沒過直接砍，不評分）

- 主題落在「飲食／餐飲／食物**及相關**」領域嗎？（**食物優先·廣義**，團隊 2026-06-11 定）——含上游（食材／農業）、中游（加工／物流）、下游（餐廳／外送／零售）、橫切面（法規／設備／行銷／永續），不是只有食物本身。AI／科技非必要、是加分不是門檻；只擋「跟飲食完全無關」的（純無人機政策、跟飲食無連結的純 AI 研究、`tail→ai` 誤判）
- 事實可查嗎？不是謠言 / 純炒作稿？
- 在這期時效窗內嗎？
- （Stream B 限定）翻得成人話嗎？翻不出人味的純技術規格 → 砍

#### 五面向評分（各 0–5，加權加總）

| 面向 | 在問什麼 | 權重 | 對應 |
|------|---------|:---:|------|
| 🎁 驚喜 / 新鮮 | 「台灣沒看過的新鮮飲食事」？來自食物本身，或 AI／科技讓食物有新可能（加分）。又一則融資稿＝低分 | ×3 | 核心調性「新鮮感與發現的驚喜」 |
| 📍 在地獨家 | 亞洲在地？英文大媒體還沒報？ | ×2.5 | 差異化 |
| 🧑 人味 | 有沒有具體的人、具體的處境？ | ×2.5 | 編輯最高原則第一條 / Mise |
| 💬 可談性 | 能勾出一個好問題、讓人想轉發或回信嗎？ | ×2 | Fumet / 傳播力 |
| 🧱 事實扎實 | 有真材實料，不是空泛吹捧？ | ×1 | Passe / 反 AI slop |

> 「事實扎實」權重低，因為它比較像**及格門檻**（低於門檻在硬閘門就砍），而非加分項。
> 權重可調——這份 rubric 就是把編輯品味寫成 AI 看得懂的規則，會隨 memory 演化。

#### 編輯路由（順便決定格式）

每篇標一個 best-fit 編輯：
- **Mise lens**（長文）：有具體的人 / 處境嗎？
- **Passe lens**（快訊）：事實夠硬、夠新嗎？
- **Fumet lens**（提問）：能引出一個好問題嗎？

| 新聞 | 判斷 |
|------|------|
| "AI Is Breaking the Recipe Blog Model. AllSpice Thinks It Can Also Save It" | ✅ 驚喜5 在地2 人味3 可談5 扎實4 → **高分**。AI 同時是兇手和救星，張力勾得出好問題 → Fumet/Mise |
| "HealthifyMe AI image recognition for Indian food" | ✅ 驚喜3 在地5 人味3 可談3 扎實4 → **中高**。印度在地是亮點 → Passe |

## 7. 最終決定：AI 提案，你拍板

**永遠不純看分數自動出刊。** 純分數解不了三件事：

1. **一期是一個「組合」，不是排行榜。** 不能 6 篇全融資、全韓國；編輯要輪替、節奏要有起伏；兩篇講同件事會互相打架。
2. **有些低分但你非選不可。** 突發、呼應你這週在做的事——策略理由只有你知道。
3. **分數只是品味的代理。** 邊界案例永遠需要人。

### 兩步決定

- **① AI 初選（提案）**：不只給排序清單，而是**組好一整期推薦**——已平衡五面向 + 組合層級（多樣性、編輯輪替、跟上期會不會太像、保鮮期）。
- **② 你終選（拍板）**：approve / 換 / 否決 / 撈庫存。**最後一定是你說了算**（你是總編，符合品牌「我們是 AI 編輯，總編是人」）。

### 每篇附的說明

- 🔢 **五面向分數拆解** — 看「它強在哪」
- 💡 **一句主理由（the hook）** —「為什麼是這篇」，例：*「AI 同時是殺死食譜部落格的兇手和救星」*
- 🧑‍🍳 **建議編輯 + 切入角度**
- 沒選上但接近的，附一句 **「為什麼這次沒選 / 進庫存」**，方便你翻盤撈庫存

> 分數告訴你「有多好」，主理由告訴你「故事是什麼、鉤子在哪」——你真正拿來決定的是後者。

## 8. 庫存：合格但沒選上的怎麼辦

20 篇合格、一期只出幾篇 → 剩下的不丟，進**有時效的庫存 backlog**：

- **帶「保鮮期」標籤**：融資稿等時效性強的，一週過期自動淘汰；概念 / 常青型可放著等下期。
- **下期重新競爭**：庫存舊合格篇跟新抓到的一起重新評分排序——你永遠選的是「當下最好的」。
- **平滑淡季**：料少的週，庫存就是緩衝，不開天窗。
- **餵其他格式**：沒選上長文的變快訊、IG carousel、月報精選——好料一篇不浪費。
- **經營訊號**：每週穩定 20 篇合格卻只出幾篇 → 該提高頻率或拉高標準，這數字值得盯。

## 9. 庫存與回饋循環

**庫存 backlog（已實作 `src/lib/backlog.ts`）**：合格沒選上的不丟，進帶保鮮期的庫存（`data/backlog.json`，預設 30 天）。每期：`prune` 丟過期 → `buildCompetitorPool` 把庫存倖存者跟本期新評分合起來排序 → 選一期 → 出刊的 `remove`、沒選上的 `upsert`（重進不續命）→ `save`。所以你永遠選「當下最好的」，而非「這批最好的」。跨期迴圈由 `scripts/test-backlog.ts` 驗證。v1 是單一 flat window；分型保鮮期（融資稿短、常青長）為下一步精修。

**三個關鍵設計**：① 保鮮期到自動淘汰；② 重進不續命（保留原 enteredAt，免得「殭屍題」永遠賴著）；③ 出刊就離場（`remove`）。

**資料怎麼存**：先存成一個人眼可讀的 JSON 檔（`data/backlog.json`，零設定），不是資料庫——現在資料小、出刊前手動跑一次就夠。`BacklogStore` 介面已抽象，之後排程自動化 / 團隊上線看，再換 Supabase，邏輯不動、只換儲存那層。`SeenStore`（去重）同理。

**回饋循環（仍待儲存層）**：你每次終選（選了哪些、否決了哪些、自己加了哪些）= 最好的訓練訊號 → 回寫三位編輯的 Soul + Memory → 初選越來越貼你的品味。Soul 已有（`docs/editors/*-soul.md`），動態 Memory 未實作。**Phase 1 盯得緊，之後放手更多，否決權永遠在你手上。**

## 10. 實作優先序

1. **修兩個 bug**（去重 `Date.now()`、`scoreRelevance` 的 `"ai"` substring）— 篩選站穩的前提
2. **建 seen store + 去重**（先 JSON 檔，後 Supabase）
3. **Opus 評估 prompt**（整池每篇：硬閘門 + 五面向 + 編輯路由 + 主理由，同一次呼叫）
5. **初選組裝**（組合層級平衡）+ 你的拍板介面
6. **庫存 backlog** + 保鮮期
7. **終選回寫 memory**
